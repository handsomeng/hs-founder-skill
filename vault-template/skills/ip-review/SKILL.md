# 审稿

## 前置

1. 读 `00-人设中枢/` 三份档案
2. 拿到稿子：他直接粘贴，或指明 `03-脚本框架/` 里哪个文件（已填完留白的版本）

## 三道检查

### 第一道：结构（对照七步说服链）

钩子 / 资格 / 痛点 / 核心观点 / 证据 / 反驳 / 行动指令，逐项核对。重点抓四个常见病：

- 钩子超过 3 秒还没给到钩住人的东西
- 一条内容塞了两个以上核心观点
- 证据段还是空的或者太抽象（没有具体人、事、数字）
- 结尾没有行动指令，或指令和商业目的对不上

### 第二道：人设一致性

- 每句话问一遍：这像他说的吗？（对照语言指纹的风格描述和原话样本）
- 是否用了禁用词、碰了红线清单
- 观点是否和观点库冲突（前后打脸要指出来）

### 第三道：去 AI 味

逐句扫这些症状，命中就标出来：

- 空词：赋能、抓手、闭环、底层逻辑、认知升级（除非语言指纹里他真这么说话）
- 三连排比、每段结尾升华、密集反问
- 「首先/其次/最后」式书面结构词出现在口播稿里
- 过度对称的句式（人不会这样说话）

## 产出格式

按严重度输出问题清单，每条：位置（引用原句）+ 问题 + 改法建议（给出改后示例句，标注「示例，按你的话改」）。

- 🔴 必须改：结构缺步、碰红线、观点打脸
- 🟡 建议改：AI 味、不像他、证据偏弱
- 🟢 顺嘴提：可选优化

最后给一个总判断：能发 / 改完再发 / 建议重写（说明原因）。

## 硬规则

- 只出清单，不主动重写整篇。他明确说「帮我改」才动笔，且改完标注每处改动
- 稿子没问题就说没问题，禁止为了显得有用硬挑毛病
- 他采纳修改并定稿后，提醒他把最终版存进 `04-成稿归档/`，文件头部记：发布平台、发布日期。发布几天后把数据（播放、点赞、评论、转化）记到 `05-数据复盘/`，同名文件
